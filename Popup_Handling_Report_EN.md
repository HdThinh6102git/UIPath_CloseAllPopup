This report outlines the detailed classification of Popups (Dialogs/Modals) encountered during the automation of insurance websites, the technical solutions implemented in the project to detect and close them automatically, and the actual implementation status of each popup type.

## Detailed Technical Solutions for Each Popup Type

### 1. DOM Popup (In-Page Modals & Overlays)

#### Technical Nature
This is the most common type of popup found on modern insurance portals. They are dynamic user interface components (Modals, Overlays, Dialogs) rendered using CSS and HTML. Upon appearance, they create a semi-transparent backdrop overlay that blocks interactions with the underlying webpage.

#### Current Status
* <span style="color:green">**Fully Handled**</span>

#### Implemented Technical Solution
We handle this popup type using a smart algorithm written in [DetectPopupAndClose.js](file:///c:/Users/PC/Desktop/QuintetCompany/projects/insurance_auto_login_ms_uipath/UI_Path_Project/UIPath_ClosePopupInUI/DetectPopupAndClose.js), which is injected via the `NInjectJsScript` activity in UiPath:

1. **Broad Selector Scanning**: Identifies all potential popup elements using an array of attribute-based selectors:
   ```javascript
   var popupSelectors = [
       '[role="dialog"]', '[aria-modal="true"]', 
       '[class*="modal" i]', '[class*="popup" i]', '[class*="overlay" i]'
   ];
   ```
2. **Visibility Check (isVisible)**: Ensures only elements that are actually visible on screen (with `width > 20px` and `height > 20px`) are processed, preventing accidental clicks on hidden elements.
3. **z-index Prioritization**: Sorts the detected popups by their CSS `z-index` in descending order, ensuring the robot always interacts with the topmost visible popup blocking the screen.
4. **Multi-Language Close Button Clicking (Korean Integrated)**: Scans for clickable elements (`button`, `a`, `role="button"`) inside the popup and compares their text against a comprehensive list of safe closing terms:
   ```javascript
   var closeWords = [
       "close", "ok", "cancel", "x", "×", "got it", "no thanks", "not now", "accept", "agree",
       "닫기", "확인", "취소", "동의", "동의함", "수락", "허용", "오늘 하루 보지 않기", "다시 보지 않기"
   ];
   ```

### 2. Native Dialog (Browser System Dialogs)

#### Technical Nature
These dialogs are spawned directly by the browser engine when executing JavaScript instructions like `window.alert()`. Since they exist at the application layer of the browser and operating system, normal webpage JavaScript is **fully frozen** and cannot interact with these dialogs.

#### Current Status
* <span style="color:green">**Fully Handled**</span>

#### Implemented Technical Solution
To bypass this limitation, the robot relies on out-of-DOM interaction mechanisms:
* **UiPath's `NClosePopup` Activity**: This modern activity interacts directly with the browser's API level to locate system buttons like `OK`, `Cancel`, or `Close` and triggers them from outside the webpage.
* **System Keyboard Shortcuts (Hardware Events)**: Sends an `Esc` key directly to the active browser to close these dialogs instantly without waiting:
  ```xml
  Shortcuts="[d(hk)][k(Esc)][u(hk)]"
  ```

---

### 3. Window Popup (Independent New Windows/Tabs)

#### Technical Nature
When clicking certain insurance links or terms, the webpage triggers an instruction to open a completely new browser window or tab (`window.open()`).

#### Current Status
* <span style="color:blue">**Modularized as a Reusable Component**</span>: The main process flow does not directly embed this logic to maintain a lightweight and clean footprint. Instead, it is packaged as an independent dedicated module. When users need to close Window Popups, they can simply invoke this custom module to complete the task.

---

## Technical Notes & Appendix: Window Popup Handling Solution (Independent Module)

When integrating or invoking this Window Popup handling module into other business workflows, the detailed operational mechanism includes:

1. **Browser Instance Management**:
   * **How it works:** Uses the `in_Browser` variable (`UiElement`) passed dynamically across workflows. The robot sticks strictly to the designated original workspace window, ignoring background spam or advertisement windows to avoid losing navigation focus.
2. **Context Switching**:
   * **How it works:** Uses the `Use Application/Browser` activity with dynamic attachment (`Attach Mode`) to switch focus from the parent window to the newly opened window. Then, dismisses the tab/window by sending a `Ctrl + W` hotkey combination (via `Keyboard Shortcuts`) or executing `window.close()` (via `Inject Js Script`) before returning control back to the main workflow.
